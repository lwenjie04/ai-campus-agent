## 0. 用户最新确认的实现约束（高优先级，覆盖申报书实现细节）

以下内容由项目当前开发要求确认，作为近期开发的实际落地方案：

- 大模型方案（现阶段）：**不做微调**，先采用“通用语言大模型 + 提示词工程（Prompt Engineering）”
- 数字人方案（现阶段）：**不做实时数字人生成/驱动**，使用你们已生成的视频素材；在回复时按规则播放特定视频
- 工程侧重点：这是一个**侧重使用 Vue3 开发**的任务，前端交互与页面能力是主要工作面

落地影响：

- 将“微调/Fine-tuning”从 MVP 主路径移到后续增强阶段
- 将“ASR/TTS/表情驱动”从 MVP 主路径移到后续增强阶段
- MVP 重点转为：`Vue3 前端体验 + Prompt 设计 + RAG 接口预留 + 视频触发播放机制`

## 1. 项目定位（后续开发总目标）

- 项目名称：数字人智能问答系统（以广二师为例）
- 场景定位：面向广东第二师范学院师生的校园问答与服务导航
- 核心价值：解决校园信息分散、跨部门咨询效率低、回复不统一的问题
- 目标形态：校园数字人 + 智能问答系统（文本为基础，逐步扩展语音/数字人交互）

## 2. 申报书明确的核心技术路线

- `Agent` 技术路线：负责对话流程管理、任务拆解、意图分析、上下文协调
- `DeepSeek` 等大模型：作为基础问答与生成能力（通用模型）
- `RAG`（检索增强生成）：接入校园专属知识库，降低幻觉，提升准确性与时效性
- 微调/优化：申报书提到 `Fine-tuning`、`Chain of Thought` 训练思路与数据优化
- 数字人集成：申报书规划包含 `ASR/TTS`、表情/动作等（属于中后期能力）

## 3. 需求范围（后续实现要对齐）

基础功能（优先级高）

- 校园常见问题问答（教务、课程、考试、流程、生活服务等）
- 多轮对话（保留上下文，持续追问）
- 校园知识库检索与引用式回答（RAG）
- 面向不同用户角色的回答适配（学生/教师等）

扩展功能（申报书规划，中后期）

- 语音输入/输出（ASR/TTS）
- 情感识别/交互优化
- 数字人形象与动作/表情联动
- 更强的推理与复杂流程引导（例如跨部门办理事项）

## 4. 数据建设要求（开发必须重视）

- 数据来源：学校官网、学生手册、规章制度、通知、流程说明、公开资料等
- 关键工作：数据收集、清洗、结构化、持续更新
- 目标：形成校园专属知识库，而不是只依赖通用大模型
- 申报书明确问题：若不做知识库和微调，通用模型无法准确覆盖校内细则

## 5. 申报书提出的重点问题（我们开发时要逐项对应）

- 数据获取与清洗难题：校园信息分散、格式不统一
- 多步推理问题：复杂办事流程问答需要步骤化回答
- 大模型幻觉问题：需使用 RAG 保证真实性
- 数据对回答准确率影响：需要持续迭代数据集
- 多轮对话连贯性：Agent 管理上下文与对话流程

## 6. 当前阶段建议的工程落地顺序（按申报书精神执行）

阶段 1：可用问答 MVP（当前代码所处阶段）

- 前端聊天界面（已开始）
- 后端 LLM 接口联调
- 基础 system prompt + 用户画像注入
- 基础错误处理、加载状态、消息渲染

阶段 2：校园知识库接入（优先推进）

- 知识库数据格式定义（文档切片、来源、更新时间）
- 检索接口（RAG）接入
- 回答附来源/依据（提升可信度）
- 管理数据更新流程（至少半自动）

阶段 3：Agent 化与复杂流程问答

- 意图分类（咨询/流程引导/信息查找）
- 多步任务编排（检索 -> 生成 -> 校验 -> 输出）
- 多轮上下文策略（短期记忆/会话摘要）

阶段 4：数字人交互能力（中后期，按当前实现约束调整）

- 优先：预生成数字人视频素材的触发播放（规则映射）
- 后续可选：TTS 语音播报
- 后续可选：ASR 语音输入
- 后续可选：实时数字人形象/动画联动

## 10. 我们后续开发的“执行约束”（记忆规则）

- 不把系统做成泛聊天机器人，必须围绕“广二师校园服务问答”场景
- 优先提升准确性与可追溯性，而不是先堆炫技功能
- 所有功能设计尽量能映射到申报书中的技术路线：`Agent + DeepSeek + RAG + 数字人`
- MVP 阶段先做文本问答打通，再做语音/数字人
- 当前 MVP 明确采用：`通用大模型 + 提示词工程`（不做微调）
- 当前数字人实现明确采用：`回复触发预生成视频播放`
- 数据与知识库建设是主线，不是附属功能
